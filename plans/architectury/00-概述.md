# ImmersiveCinematics — Architectury 架构迁移计划

## 概述

将旧版 Forge-only 单项目架构迁移到 **Architectury 多平台架构**（common + fabric + forge），使模组同时支持 Fabric 和 Forge 加载器。

## 架构现状

### 旧架构（`old/` 目录）

- **113 个 Java 文件**，全部在 `com.immersivecinematics.immersive_cinematics` 包下
- **15 个文件**含 Forge 特定导入（`net.minecraftforge.*`）
- **98 个纯 Vanilla 文件**（可直接移入 common 模块）
- 单一 `build.gradle` + Forge `@Mod` 入口

### 新架构（当前已搭建）

```
immersive_cinematics/
├── common/          ← 跨平台共享代码
│   ├── src/main/java/.../ImmersiveCinematics.java   (stub)
│   └── src/main/resources/immersive_cinematics.mixins.json
├── fabric/          ← Fabric 入口点
│   └── src/main/java/.../fabric/
│       ├── ImmersiveCinematicsFabric.java
│       └── client/ImmersiveCinematicsFabricClient.java
├── forge/           ← Forge 入口点
│   └── src/main/java/.../forge/ImmersiveCinematicsForge.java
└── 根目录 build.gradle + settings.gradle  (Architectury Loom)
```

## 迁移策略

### 核心原则

1. **一个模块一个模块迁移** — 每次只迁移一个功能模块
2. **每次迁移后编译 + 进游戏测试** — 确保不引入回归
3. **尽量少改逻辑** — 只替换平台 API 调用，不重写业务逻辑
4. **不引入新依赖** — 除非 Architectury API 无法覆盖的功能
5. **Forge 特有事件留 forge 模块** — 不需要强行把所有事件抽象到 common（参考 FTB-Quests）

### 参考项目

FTB-Quests（`example/FTB-Quests-1.20.1-main/`）提供了以下可复用的模式：

| 模式 | 用途 | 参考文件 |
|------|------|---------|
| `SimpleNetworkManager` + `BaseS2CMessage`/`BaseC2SMessage` | 网络层，比原始 `NetworkManager` 更简洁 | `common/.../net/FTBQuestsNetHandler.java` |
| `@ExpectPlatform` | 平台差异委托（fabric/forge 各自实现） | `common/.../net/FTBQuestsNetHandler.java` + `fabric/.../net/FTBQuestsNetHandlerImpl.java` |
| `EnvExecutor.runInEnv(Env.CLIENT, ...)` | 在 common 代码中安全执行仅客户端初始化 | `common/.../FTBQuests.java` |
| `EnvExecutor.getEnvSpecific(() -> proxy, () -> proxy)` | 通用客户端/服务端代理模式 | `common/.../FTBQuests.java` |
| Forge 自有事件直接在 forge/ 模块注册 | 不强行抽象到 common | `forge/.../FTBQuestsForge.java` |

### 模块迁移模式

每个模块的迁移流程：

```
1. 在 common/ 下创建模块目录结构
2. 复制旧代码到 common/
3. 替换 Forge 特定 API 为 Architectury API
4. 将平台特定代码（如 config 加载）移到 forge/ 或 fabric/
5. 更新 build.gradle 或 mixins.json（如需要）
6. 编译测试
7. 进游戏测试
```

### Forge API → Architectury API 映射

| Forge API | Architectury API |
|-----------|-----------------|
| `@Mod`, `Mod.EventBusSubscriber` | `ImmersiveCinematics.init()` + 直接事件注册 |
| `MinecraftForge.EVENT_BUS` | 各事件类的 `.register()` 方法 |
| `ForgeConfigSpec` | 自定义跨平台配置方案 |
| `ServerStartedEvent` | `LifecycleEvent.SERVER_STARTED` |
| `ServerStoppingEvent` | `LifecycleEvent.SERVER_STOPPING` |
| `TickEvent.ServerTickEvent` | `TickEvent.SERVER_POST` |
| `TickEvent.ClientTickEvent` | `ClientTickEvent.CLIENT_POST` |
| `PlayerEvent.PlayerLoggedInEvent` | `PlayerEvent.PLAYER_JOIN` |
| `PlayerEvent.PlayerLoggedOutEvent` | `PlayerEvent.PLAYER_QUIT` |
| `RegisterCommandsEvent` | `CommandRegistrationEvent.EVENT` |
| `RenderGuiOverlayEvent.Pre/Post` | `ClientGuiEvent.RENDER_HUD` |
| `RegisterGuiOverlaysEvent` | `ClientGuiEvent.RENDER_HUD` |
| `RegisterKeyMappingsEvent` | Architectury 客户端注册 |
| `ViewportEvent.ComputeCameraAngles` | 保留 Mixin 或使用 Architectury Hook |
| `FMLClientSetupEvent` | `ClientLifecycleEvent.CLIENT_SETUP` |
| `FMLCommonSetupEvent` | 直接在 `init()` 中初始化 |
| `NetworkRegistry/SimpleChannel` | `SimpleNetworkManager` + `BaseS2CMessage`/`BaseC2SMessage` |
| `ForgeRegistries` | `BuiltInRegistries`（Vanilla） |
| `ServerLifecycleHooks` | 通过 `LifecycleEvent` 获取 Server 引用 |
| `ConfigScreenHandler` | 平台各自实现（`@ExpectPlatform`） |
| `Dist` / `FMLEnvironment.dist` | `Env` / `EnvExecutor` |
| `@ExpectPlatform` | Architectury 内置，fabric/forge 分别实现 |
| `DistExecutor` | `EnvExecutor.runInEnv(Env.CLIENT, ...)`（Architectury 提供） |
| `InputEvent.Key` / `MouseButton` | `ClientRawInputEvent` 或 Mixin |

## 通用注意事项

1. **包路径不变** — 所有代码保持 `com.immersivecinematics.immersive_cinematics` 包路径，方便跨模块引用
2. **mixin 统一管理** — mixin 类和 `mixin.json` 放在 `common/`，两个加载器都使用
3. **平台代码分离** — `fabric/` 和 `forge/` 只放入口点和平台特有适配代码
4. **平台差异处理**（个人作者 + AI 开发模式）
   - **`@ExpectPlatform` 不用** —— AI 不理解 "throw 等另一边补" 的模式
   - 需要平台差异的地方，直接在 `forge/` 或 `fabric/` 里写死，common 不引用
   - Forge 特有事件直接注册在 `ImmersiveCinematicsForge.java`，Fabric 特有代码直接写在 `ImmersiveCinematicsFabric.java`
   - 两个平台都要用但实现不同的功能，在 common 定义接口，forge/fabric 各自实现
6. **资源文件** — lang、textures、pack.mcmeta 放 `common/` 的 resources 下
7. **editor 开关** — `EDITOR_ENABLED` 常量保留，控制编辑器编译与否
8. **每个阶段都是一个可编译+可运行的状态** — 即使功能不完整，游戏必须能正常启动
